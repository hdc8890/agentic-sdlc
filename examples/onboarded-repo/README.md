# Example: Onboarded Repository

This example shows what an onboarded team repository looks like after running `agentic init` and `agentic profile --generate`.

## Structure

```
your-repo/
  .agentic/
    profile.json    # Describes the repo to the orchestration engine
    policy.json     # Controls team autonomy level and review requirements
```

These two files are the only things a team needs to add to their repo to join the platform.

## profile.json

The repo profile describes the repository to the orchestration engine:

- **ecosystems** — what languages/runtimes are in use (detected automatically)
- **working_roots** — directories the orchestrator may work in
- **protected_paths** — paths that always require human review before modification
- **commands** — how to run tests, lint, and build (detected from package.json, Cargo.toml, etc.)
- **ci / pull_request** — CI checks and approval requirements

The profile is generated by `agentic profile --generate` and should be reviewed and committed. It stays in sync with the repo — re-run `agentic profile --generate` when you add new ecosystems or change CI.

## policy.json

The team policy controls how autonomously the platform acts in this repository:

- **autonomy_level** — the default for all tasks (start with `semi_autonomous` for most teams)
- **require_human_review_on** — conditions that always pause for review regardless of autonomy level
- **auto_merge** — whether the platform may merge PRs automatically when all checks pass
- **task_type_overrides** — per-task-type autonomy overrides (e.g., let `docs` tasks run more autonomously)
- **notification_channels** — where to send status updates

## How these files are used

```
Team repo                  Platform services
─────────────────────      ──────────────────────────────
.agentic/profile.json ──►  Orchestration engine uses to:
.agentic/policy.json  ──►  • know which commands to run
                           • respect protected paths
                           • enforce autonomy level
                           • collect evaluation evidence
                           • determine when to pause for review
```

## Onboarding a new repo

```bash
# 1. Scaffold .agentic/
agentic init

# 2. Auto-detect repo context
agentic profile --generate

# 3. Review the generated files
cat .agentic/profile.json
cat .agentic/policy.json

# 4. Validate before committing
agentic validate

# 5. Commit
git add .agentic/
git commit -m "chore: onboard to Agentic SDLC framework"
```

Or use the GitHub Action for CI-driven onboarding — see [action/README.md](../../action/README.md).
